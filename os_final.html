CH.01 Giới Thiệu Hệ Điều Hành ▶
Hệ điều hành là gì? Vai trò đối với phần mềm ứng dụng?
▶
Định nghĩa: HĐH là phần mềm trung gian giữa phần cứng và người dùng/ứng dụng.
  • Với người dùng: cung cấp giao diện thuận tiện (CLI, GUI, Touch)
  • Với ứng dụng: cung cấp môi trường thực thi, truy cập tài nguyên qua System Call thay vì trực tiếp phần cứng
  • Với phần cứng: quản lý CPU, RAM, I/O, file
3 mục tiêu HĐH: (1) Tiện lợi người dùng  (2) Hiệu quả tài nguyên  (3) Khả năng mở rộng
⭐ Chat về chủ đề này
Nhân (Kernel) là gì? Hai chế độ hoạt động thông dụng nhất?
▶
Kernel: phần lõi của HĐH, luôn chạy trong bộ nhớ, kiểm soát toàn bộ phần cứng, chạy ở Kernel Mode.
  • User Mode — ứng dụng chạy ở đây, quyền hạn chế, KHÔNG thể trực tiếp truy cập phần cứng
  • Kernel Mode — HĐH chạy ở đây, toàn quyền truy cập phần cứng
Chuyển chế độ: từ User sang Kernel qua System Call hoặc Ngắt (Interrupt).
Lệnh đặc quyền (Privileged Instruction): chỉ chạy được ở Kernel Mode. Nếu User Mode cố chạy → trap → HĐH bắt lỗi và chấm dứt tiến trình.
⭐ Chat về chủ đề này
Ngắt (Interrupt) là gì? Cơ chế hoạt động?
▶
Ngắt: tín hiệu từ phần cứng (hoặc phần mềm) báo CPU có sự kiện cần xử lý ngay.
  • CPU dừng công việc hiện tại, lưu trạng thái (PC, thanh ghi)
  • Chạy interrupt handler (trình xử lý ngắt)
  • Sau khi xử lý xong → tiếp tục công việc cũ
Tại sao quan trọng? Ngắt là cơ chế nền tảng của: I/O bất đồng bộ, đa nhiệm, bảo vệ CPU timeout. Không có ngắt → CPU phải polling liên tục.
⭐ Chat về chủ đề này
Ảo hoá (Virtualization) là gì? Máy ảo dùng phương pháp nào?
▶
Ảo hoá: kỹ thuật tạo ra phiên bản ảo của tài nguyên (CPU, RAM, thiết bị).
  • Máy ảo (VM) chạy HĐH riêng trên cùng phần cứng vật lý
  • Mỗi VM "nghĩ" nó sở hữu toàn bộ phần cứng
  • Phương pháp: Abstraction / Virtualization
Ví dụ: VMware, VirtualBox — chạy nhiều OS trên 1 máy vật lý.
⭐ Chat về chủ đề này
Mối quan hệ Đa nhiệm (Multitasking) và Đa chương (Multiprogramming)?
▶
  • Đa chương (Multiprogramming): nhiều chương trình trong RAM cùng lúc. CPU không idle — nếu 1 tiến trình chờ I/O, CPU chuyển sang cái khác.
  • Đa nhiệm (Multitasking): CPU chuyển đổi rất nhanh giữa tiến trình (time-sharing) → cảm giác đồng thời.
Mối quan hệ: Đa nhiệm là mở rộng của đa chương. Đa chương ưu tiên CPU utilization; đa nhiệm ưu tiên response time.
Bộ nhớ chính đa nhiệm chứa: Kernel HĐH + tất cả tiến trình đang chạy. Lệnh phải nằm trong RAM để CPU thực thi được.
⭐ Chat về chủ đề này
Real-time OS đặc điểm gì? Môi trường Client-Server có đặc trưng gì?
▶
HĐH thời gian thực (Real-time OS):
  • Đáp ứng sự kiện trong thời hạn cố định (deadline)
  • Hard real-time: deadline tuyệt đối; Soft real-time: linh hoạt hơn
  • Khó khăn lập trình: đảm bảo không bao giờ trễ deadline
Client-Server:
  • Client gửi yêu cầu (request) → Server xử lý → trả response
  • Đặc trưng: phân tán, 1 server phục vụ nhiều client

GNU là gì? GPL là gì? Quản lý bộ nhớ chính gồm những công việc nào?
▶
  • GNU: dự án phần mềm tự do (Richard Stallman — "GNU's Not Unix")
  • GPL (GNU General Public License): giấy phép nguồn mở — ai sử dụng phải chia sẻ lại mã nguồn
Quản lý bộ nhớ chính:
  • Theo dõi vùng nhớ đang dùng / còn trống
  • Cấp phát bộ nhớ cho tiến trình khi cần
  • Giải phóng khi tiến trình kết thúc
  • Quyết định tiến trình nào được nạp vào RAM
"Người dùng định danh bằng ID+mật khẩu, có hoặc không quyền chạy chương trình" → mô tả nhiệm vụ của Security Manager (bộ quản lý bảo mật).

CH.02 Cấu Trúc Hệ Điều Hành ▶
System Call là gì? Lập trình viên tiếp cận qua phương tiện gì? API của Linux là thư viện nào?
▶
System Call: giao diện lập trình giữa ứng dụng (User Mode) và kernel (Kernel Mode). Cách duy nhất User Mode yêu cầu dịch vụ kernel.
  • Lập trình viên tiếp cận qua API (Application Programming Interface)
  • API Linux = thư viện glibc (GNU C Library)
  • Người dùng bình thường: tiếp cận qua Shell (CLI/GUI/Touch)
5 nhóm System Call: (1) Quản lý tiến trình: fork/exec/exit/wait  (2) Quản lý file: open/read/write/close  (3) Quản lý thiết bị  (4) Thông tin hệ thống  (5) Giao tiếp: pipe/socket
⭐ Chat về chủ đề này
CLI/GUI/Touch gọi chung là gì? Linker là gì? Ứng dụng lỗi tạo tập tin nào?
▶
  • CLI/GUI/Touch → gọi chung là Shell hoặc User Interface
  • Linker: kết hợp nhiều object files thành 1 executable
  • Loader: nạp executable vào RAM để thực thi
  • Ứng dụng crash tạo core dump file — lưu toàn bộ bộ nhớ tiến trình tại thời điểm lỗi
Đối tượng sử dụng dịch vụ HĐH: người dùng (qua Shell) và lập trình viên (qua API).
⭐ Chat về chủ đề này
So sánh 3 kiến trúc HĐH: Monolithic, Layered, Microkernel — lợi/khuyết?
▶
1. Monolithic (Nguyên khối):
  • Lợi: hiệu năng cao, gọi hàm trực tiếp trong kernel, không overhead
  • Khuyết: khó bảo trì; 1 lỗi có thể sập toàn bộ hệ thống
  • Ví dụ: Linux, Unix
2. Layered (Phân lớp):
  • Lợi: dễ debug từng lớp; mỗi lớp chỉ dùng dịch vụ lớp dưới
  • Khuyết: hiệu năng kém — phải đi qua nhiều lớp (overhead)
3. Microkernel:
  • Kernel nhỏ tối thiểu; hầu hết dịch vụ chạy ở User Space
  • Giao tiếp qua Message Passing
  • Lợi: ổn định, bảo mật  Khuyết: chậm hơn do overhead message passing
Câu hỏi thi: "Phương pháp cung cấp hầu hết dịch vụ dưới dạng ứng dụng người dùng và giao tiếp bằng truyền thông điệp?" → Microkernel
⭐ Chat về chủ đề này
Mục tiêu phân chia thời gian CPU? Nhân HĐH có vai trò gì? Máy ảo dùng phương pháp nào?
▶
Phân chia thời gian CPU (Time-sharing): cho phép nhiều người dùng/tiến trình dùng CPU công bằng và hiệu quả. Tạo ảo giác mỗi người có CPU riêng. Nhân (Kernel): kiểm soát và quản lý toàn bộ hệ thống; cầu nối giữa phần cứng và phần mềm; thực thi ở Kernel Mode. Máy ảo: dùng phương pháp Abstraction / Virtualization. Mỗi VM nghĩ nó sở hữu toàn bộ phần cứng.

CH.03 Tiến Trình & Liên Lạc (IPC) ▶
Tiến trình (Process) là gì? Bố cục bộ nhớ gồm những gì? 5 trạng thái?
▶
Tiến trình: chương trình đang được thực thi. Trở thành tiến trình khi được nạp vào RAM và bắt đầu chạy. Bố cục bộ nhớ:
  • Text: mã lệnh (read-only)
  • Data: biến toàn cục và static
  • Heap: cấp phát động (malloc/new) — tăng lên trên
  • Stack: biến cục bộ, tham số hàm — tăng xuống dưới
5 trạng thái: New → Ready → Running → Waiting → Terminated
Ready→Running (scheduler chọn) | Running→Waiting (chờ I/O) | Waiting→Ready (I/O xong)
❌ Không tồn tại: Waiting → Running trực tiếp
⭐ Chat về chủ đề này
PCB là gì? Context Switch xảy ra khi nào? Bước đầu tiên phải làm gì?
▶
PCB (Process Control Block): cấu trúc dữ liệu lưu toàn bộ thông tin tiến trình: PID, trạng thái, PC, thanh ghi CPU, thông tin bộ nhớ, I/O, lập lịch. Hàng đợi dùng linked list. Context Switch:
  • Bước đầu tiên: lưu PCB của tiến trình đang chạy → nạp PCB tiến trình tiếp theo
  • Xảy ra khi: ngắt, system call, time quantum hết
  • Mục tiêu: cho phép đa nhiệm
  • Hạn chế: tốn thời gian, trong khi chuyển CPU không làm được việc hữu ích (pure overhead)
⭐ Chat về chủ đề này
fork() trong UNIX: giá trị trả về? Tiến trình con thừa kế gì? Công thức tính số tiến trình?
▶
fork(): tạo tiến trình con bằng cách nhân đôi tiến trình cha. Giá trị trả về:
  • 0 → trong tiến trình con
  • PID của con → trong tiến trình cha
  • -1 → lỗi
Tiến trình con thừa kế từ cha: code, data, file descriptors, environment. Không thừa kế: memory locks, pending signals (có PID riêng).
Công thức: n lần fork() không có điều kiện → tạo ra 2ⁿ tiến trình (bao gồm tiến trình gốc). Ví dụ: 3 fork = 8 tiến trình.
⭐ Chat về chủ đề này
IPC là gì? 2 phương pháp? Tối thiểu phải có thao tác nào?
▶
IPC = Inter-Process Communication (Giao tiếp liên tiến trình).
  • Shared Memory: các tiến trình đọc/ghi vùng nhớ chung. Nhanh, cùng máy. Cần đồng bộ hoá.
  • Message Passing: gửi/nhận thông điệp qua kernel. Dễ dùng, phù hợp hệ thống phân tán.
Tối thiểu phải có: send() và receive().
2 hình thức giao tiếp server-client phổ biến: Sockets và Remote Procedure Calls (RPCs)
⭐ Chat về chủ đề này
Direct vs Indirect communication? Non-blocking send? Zero capacity queue? Pipe là gì?
▶
Direct: P gửi thẳng cho Q — phải biết tên/PID nhau. Mỗi cặp có đúng 1 link. Indirect: qua mailbox/port — P và Q không cần biết nhau. Nhiều tiến trình dùng cùng mailbox được. Non-blocking send: người gửi không chờ, tiếp tục ngay sau khi gửi (asynchronous). Zero capacity queue:
  • Không có buffer → người gửi phải chặn (block) cho đến khi người nhận sẵn sàng
  • Gọi là rendezvous
  • Dùng trong real-time và synchronous IPC
Pipe:
  • Ordinary pipe: 1 chiều, cha-con
  • Named pipe (FIFO): 2 chiều, không cần quan hệ cha-con
Dung lượng có hạn/vô hạn là tính chất của hàng đợi thông điệp (message queue).

CH.04 Tiểu Trình & Đồng Thời (Threads) ▶
Thread là gì? Chia sẻ gì với các thread khác? Dữ liệu riêng là gì?
▶
Thread (Tiểu trình/Luồng): đơn vị thực thi nhỏ nhất bên trong một tiến trình. Chia sẻ giữa các thread cùng process: Code (Text), Data, Heap, File descriptors. Dữ liệu RIÊNG của mỗi thread:
  • Stack (biến cục bộ, tham số)
  • Thanh ghi CPU
  • Program Counter (PC)
  • Thread ID
Thư viện đa luồng Linux: POSIX Pthreads (pthread.h).
Lợi: tạo nhanh hơn process, chia sẻ tài nguyên dễ, tận dụng đa nhân, responsive.
Khuyết: 1 thread lỗi → sập cả process; race condition, deadlock phức tạp.
⭐ Chat về chủ đề này
3 mô hình ánh xạ thread? Mô hình nào phổ biến nhất? Mô hình nào KHÔNG tồn tại?
▶
1. Many-to-One: nhiều user thread → 1 kernel thread. Nhanh nhưng 1 block = tất cả block, không tận dụng đa nhân. 2. One-to-One ⭐ (phổ biến nhất):
  • 1 user thread → 1 kernel thread. Dùng trong Windows và Linux.
  • Khuyết: tạo nhiều kernel thread → tốn tài nguyên kernel
3. Many-to-Many: linh hoạt nhất; phức tạp nhất.
KHÔNG TỒN TẠI: mô hình One-to-Many — câu bẫy kinh điển trong đề thi!
⭐ Chat về chủ đề này
Parallelism vs Concurrency? Data parallelism? Định luật Amdahl tính thế nào?
▶
  • Parallelism (Song song): thực sự chạy cùng lúc trên nhiều nhân CPU. Cần phần cứng đa nhân.
  • Concurrency (Đồng thời): luân phiên nhanh trên 1 nhân, cảm giác đồng thời.
  • Data parallelism: phân chia dữ liệu thành nhiều phần, cùng thao tác trên mỗi phần (ví dụ: nhân ma trận, merge sort).
Định luật Amdahl: Speedup = 1 / ((1-S) + S/N)  |  S = phần song song, N = số nhân

Ví dụ 1: S=0.75, N=2: 1 / (0.25 + 0.375) = 1/0.625 = 1.6×
Ví dụ 2: S=0.75, N=4: 1 / (0.25 + 0.1875) = 1/0.4375 ≈ 2.29×
⭐ Chat về chủ đề này
Deferred cancellation là gì? Bài toán nào không thể áp dụng đa luồng?
▶
Deferred Cancellation: thread bị đánh dấu để huỷ nhưng không kết thúc ngay. Thực sự kết thúc tại cancellation point (điểm an toàn) tiếp theo — tránh để lại tài nguyên chưa giải phóng. Bài toán KHÔNG thể đa luồng: có sequential dependency — mỗi bước phụ thuộc kết quả bước trước.
  • Ví dụ: n! = n×(n-1)×... (phải tuần tự)
  • CÓ THỂ đa luồng: Merge Sort (chia song song), nhân ma trận (mỗi phần tử độc lập)

CH.05A Lập Lịch CPU Cơ Bản ▶
Lập lịch CPU là gì? Dispatcher làm gì? 5 tiêu chí đánh giá bộ lập lịch?
▶
CPU Scheduling: chọn tiến trình từ Ready Queue để cấp CPU. Chỉ tiến trình ở trạng thái Ready mới được xem xét. Tiến trình là chu kỳ xen kẽ: CPU Burst (tính toán) + I/O Burst (chờ I/O). Dispatcher: thực hiện chuyển CPU sang tiến trình được chọn. Chạy ở Kernel Mode. Công việc: context switch + chuyển User Mode + nhảy đến PC. 5 tiêu chí:
  • CPU Utilization: tối đa (max)
  • Throughput: số tiến trình/giây (max)
  • Turnaround time = Completion − Arrival (min)
  • Waiting time = tổng thời gian trong Ready Queue (min)
  • Response time = thời gian từ yêu cầu đến phản hồi đầu tiên (min)
⭐ Chat về chủ đề này
Preemptive và Non-preemptive: khi nào CPU bị lấy lại? Khi nào tiến trình trả CPU?
▶
Preemptive (Chiếm quyền): HĐH cưỡng chế lấy CPU khi:
  • Ngắt (interrupt) xảy ra
  • Time quantum hết
  • Tiến trình mới có ưu tiên cao hơn tiến trình đang chạy
Non-preemptive (Không chiếm quyền): tiến trình tự nguyện trả CPU khi:
  • Tiến trình kết thúc
  • Tiến trình chuyển sang Waiting (chờ I/O)
"Giữ quyền ưu tiên (Preemptive) là khả năng gì?" → khả năng lấy CPU từ tiến trình đang chạy để cấp cho tiến trình khác.
⭐ Chat về chủ đề này
4 giải thuật lập lịch: FCFS, SJF, Priority, Round Robin — hạn chế từng loại?
▶
FCFS (First-Come First-Served): không chiếm quyền, đơn giản. Hạn chế: Convoy Effect — tiến trình ngắn chờ tiến trình dài. SJF (Shortest Job First): ưu tiên tiến trình ngắn nhất. Lợi: tối ưu waiting time trung bình. Hạn chế: Starvation + khó dự đoán CPU burst tiếp theo. Priority Scheduling: Hạn chế: Starvation. Giải pháp: Aging — tăng dần ưu tiên theo thời gian chờ. Round Robin (RR): mỗi tiến trình chạy tối đa quantum q.
  • Lợi: response time tốt, công bằng
  • Hạn chế: q quá nhỏ → quá nhiều context switch (overhead); q quá lớn → degrades to FCFS
Công thức RR: n tiến trình, quantum q → thời gian chờ giữa 2 lần vào CPU liên tiếp = (n−1) × q
⭐ Chat về chủ đề này
Bài tập tính FCFS Waiting Time và Turnaround Time?
▶
Công thức: WT = TAT − CPU Burst  |  TAT = Completion − Arrival  |  Avg = Tổng / n
Ví dụ 1 (Avg Waiting Time):
P1=15ms, P2=6ms, P3=20ms (arrival=0, FCFS)
WT(P1)=0  |  WT(P2)=15  |  WT(P3)=15+6=21
Avg WT = (0+15+21)/3 = 12ms

Ví dụ 2 (Avg Turnaround Time):
P1=24ms, P2=6ms, P3=10ms (arrival=0, FCFS)
Completion: P1=24, P2=30, P3=40
Avg TAT = (24+30+40)/3 = 31.33ms
